# 微服務分類使用規則

## 核心原則

### 1. 模組資料隔離
- 每個微服務模組**只能查詢屬於自己模組的資料表**
- 嚴禁跨模組直接查詢資料庫（如：`project-skill-service` 不可查詢 `users` 表）
- 跨服務資料存取一律透過 **Feign Client** 呼叫對應服務的 API

### 2. 層級職責劃分

```
Controller → Service → DataAccess → Repository → JPA
```

- **Controller 層**：請求接收與回應
- **Service 層**：業務邏輯，**禁止直接使用 `EntityManager`**
- **DataAccess 層**：資料存取抽象，可使用 `EntityManager` 作內部實作
- **Repository 層**：Spring Data JPA 介面

### 3. 跨服務關聯處理

#### 外鍵參考（FK Reference）
- 需要在本地實體設定跨服務 Entity 的外鍵時，使用 `new Entity()` + `setId()` 方式：

```java
// 正確：只設定 ID，不查詢其他服務的資料表
public void mock(){

User user = new User();
user.setId(userId);
userProject.setUser(user);
}
        
```

- 此方式不會觸發資料庫查詢，僅提供 JPA 所需的 FK 值

#### 資料驗證
- 驗證跨服務 Entity 是否存在時，使用 Feign Client：

```java
public void mock() {
    if (!userServiceFeignClient.existsUserById(userId)) {
        throw new IllegalArgumentException("User not found");
    }
}
```

### 4. 使用 EntityManager 的限制

| 層級 | 是否允許 | 說明 |
|------|----------|------|
| Service | ❌ 禁止 | 業務邏輯不應直接操作持久層 |
| DataAccess | ✅ 允許 | 可作為內部實作細節使用 |
| Repository | ✅ 允許 | Spring Data JPA 底層 |

### 5. 實體共用規則
- 共用 Entity 定義放在 `backend-common` 模組
- Entity 僅供 JPA 映射參考，**不應作為跨服務資料傳輸載體**
- 跨服務資料傳輸使用 DTO/VO 與 Feign Client

### 6. JPA 關係設定
- 跨服務的 `@ManyToOne` / `@OneToMany` 關係不設定 `cascade`
- EAGER fetch 應謹慎使用，避免觸發跨服務資料表查詢
